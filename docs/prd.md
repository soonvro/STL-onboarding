# Q&A 알림 서비스 PRD

## 1. 문서 개요

### 1.1 배경
문의 접수 이후 저장, 관리자 알림, 처리 상태 관리, 완료 안내까지의 흐름을 수동으로 처리하면 누락과 중복이 발생하기 쉽다. 본 문서는 문의 등록과 처리 완료 알림을 자동화하는 간단한 Q&A 알림 서비스의 제품 요구사항을 정의한다.

### 1.2 목표
- 사용자가 웹에서 문의를 등록할 수 있다.
- 등록된 문의는 자동으로 저장되고 관리자에게 즉시 알림이 전달된다.
- 관리자는 문의 상태를 조회하고 `등록됨`, `처리중`, `완료됨`으로 관리할 수 있다.
- `완료됨` 처리 시 문의자와 관리자 모두에게 완료 알림이 전달된다.
- 동일한 `이름 + 제목` 조합의 중복 문의는 등록되지 않는다.

### 1.3 성공 기준
- 사용자는 문의 등록 성공 또는 실패 결과를 즉시 확인할 수 있다.
- 관리자는 관리자 페이지에서 전체 문의 목록과 현재 상태를 확인할 수 있다.
- 완료 처리 이후 Notion DB와 알림 메일 상태가 일관되게 반영된다.
- 입력값 검증과 중복 방지로 잘못된 데이터 유입을 줄인다.

## 2. 사용자 및 범위

### 2.1 주요 사용자
- 문의자: 공개 문의 폼을 통해 문의를 등록하는 사용자
- 관리자: 문의 목록을 조회하고 상태 및 처리 결과를 관리하는 운영자

### 2.2 포함 범위
- 공개 문의 등록 페이지
- 관리자용 문의 목록/상태 관리 페이지
- 문의 등록 시 n8n 등록 워크플로우 호출
- 완료 처리 시 n8n 완료 워크플로우 호출
- Notion DB 저장 및 상태 업데이트 연동
- 관리자 및 문의자 메일 알림
- 전화번호 필드 추가 및 기본 유효성 검사
- 동시 요청 상황까지 고려한 중복 등록 방지

### 2.3 제외 범위
- 문의자의 별도 로그인 및 본인 문의 상태 조회 기능
- 복잡한 관리자 인증 체계
- 메일 외의 알림 채널 연동
- 첨부파일 업로드

## 3. 제품 원칙 및 가정

- 문의 등록은 로그인 없이 누구나 가능하다.
- 관리자 페이지는 복잡한 인증 대신 최소 비밀번호 보호 수준의 접근 제어를 전제로 한다.
- `처리중` 상태 변경은 내부 상태 및 저장소 반영까지만 수행하며 별도 외부 알림은 발생시키지 않는다.
- `완료됨` 상태 변경 시에만 완료 처리 워크플로우를 호출한다.
- 입력 검증은 일반적인 실무 수준의 기본 검증을 목표로 한다.

## 4. 사용자 흐름

### 4.1 문의 등록 흐름
1. 문의자가 문의 페이지에서 `이름`, `이메일`, `전화번호`, `제목`, `본문`을 입력한다.
2. 시스템은 필수값, 공백-only 입력, 이메일 형식, 전화번호 형식을 검증한다.
3. 문의자가 등록 버튼을 클릭하면 시스템은 등록 워크플로우를 호출한다.
4. 등록 워크플로우는 중복 여부를 확인한다.
5. 중복이 아니면 문의를 Notion DB에 저장하고 관리자에게 메일을 전송한다.
6. 시스템은 사용자에게 등록 결과를 표시한다.

### 4.2 관리자 처리 흐름
1. 관리자는 관리자 페이지에서 문의 목록을 조회한다.
2. 각 문의의 현재 상태와 핵심 정보를 확인한다.
3. 관리자는 문의 상태를 `등록됨`, `처리중`, `완료됨` 중 하나로 변경한다.
4. `처리중` 변경 시 시스템은 상태를 저장소에 반영한다.
5. `완료됨` 변경 시 시스템은 처리 결과와 함께 완료 처리 워크플로우를 호출한다.

### 4.3 완료 처리 흐름
1. 완료 처리 워크플로우는 문의자에게 결과 메일을 전송한다.
2. 완료 처리 워크플로우는 관리자에게 완료 알림 메일을 전송한다.
3. Notion DB의 상태와 처리 결과를 `완료됨` 기준으로 업데이트한다.

## 5. 기능 요구사항

### 5.1 문의 등록 페이지
- 사용자는 `이름`, `이메일`, `전화번호`, `제목`, `본문`을 입력할 수 있어야 한다.
- 각 필드는 제출 전에 검증되어야 한다.
- 등록 버튼 클릭 시 중복 제출을 방지할 수 있어야 한다.
- 등록 처리 중에는 진행 중 상태를 사용자에게 알려야 한다.
- 등록 성공 시 성공 메시지를 표시해야 한다.
- 등록 실패 시 실패 원인을 이해할 수 있는 메시지를 표시해야 한다.

### 5.2 입력값 검증
- `이름`, `이메일`, `전화번호`, `제목`, `본문`은 필수 입력값이다.
- 공백만 입력한 경우 유효하지 않은 값으로 처리해야 한다.
- 이메일은 일반적인 이메일 형식 검사를 통과해야 한다.
- 전화번호는 일반적인 전화번호 형식 검사를 통과해야 한다.
- 제목과 본문은 의미 없는 빈 문자열로 저장되지 않아야 한다.

### 5.3 관리자 페이지
- 관리자는 문의 목록을 최신 상태로 조회할 수 있어야 한다.
- 목록에서 각 문의의 `이름`, `이메일`, `전화번호`, `제목`, `상태`, `등록 시각`을 확인할 수 있어야 한다.
- 관리자는 개별 문의의 상세 내용과 처리 결과를 확인하거나 입력할 수 있어야 한다.
- 관리자는 문의 상태를 변경할 수 있어야 한다.
- 상태 변경 결과는 화면에 즉시 반영되어야 한다.

### 5.4 중복 등록 방지
- 동일한 `이름 + 제목` 조합은 중복 문의로 판단한다.
- 중복 문의는 저장되지 않아야 한다.
- 사용자는 중복 등록 실패 사실을 안내받아야 한다.
- Notion DB 응답이 느린 상황에서도 동시에 같은 문의가 여러 번 저장되지 않도록 설계해야 한다.

### 5.5 외부 워크플로우 연동
- 문의 등록 시 n8n 등록 워크플로우가 호출되어야 한다.
- 등록 워크플로우는 Notion DB 저장과 관리자 메일 전송을 수행해야 한다.
- 문의 완료 시 n8n 완료 처리 워크플로우가 호출되어야 한다.
- 완료 처리 워크플로우는 문의자 결과 메일, 관리자 완료 메일, Notion 상태 및 처리 결과 업데이트를 수행해야 한다.

## 6. 데이터 및 상태 정의

### 6.1 문의 데이터 항목
문의는 최소 아래 속성을 가진다.

| 항목 | 설명 |
| --- | --- |
| `id` | 문의 식별자 |
| `name` | 문의자 이름 |
| `email` | 문의자 이메일 |
| `phone` | 문의자 전화번호 |
| `title` | 문의 제목 |
| `body` | 문의 본문 |
| `status` | 문의 상태 |
| `resolution` | 처리 결과 또는 답변 내용 |
| `createdAt` | 생성 시각 |
| `updatedAt` | 마지막 수정 시각 |

### 6.2 상태 정의

| 상태 | 설명 |
| --- | --- |
| `등록됨` | 문의가 정상 접수된 초기 상태 |
| `처리중` | 관리자가 확인 후 처리 중인 상태 |
| `완료됨` | 처리 결과가 확정되어 완료 워크플로우가 실행되는 상태 |

### 6.3 외부 연동 이벤트

| 이벤트 | 발생 시점 | 목적 |
| --- | --- | --- |
| `문의 등록 요청` | 사용자가 문의 등록 시 | 중복 확인, Notion 저장, 관리자 메일 전송 |
| `완료 처리 요청` | 관리자가 상태를 `완료됨`으로 변경 시 | 문의자 메일 전송, 관리자 완료 알림, Notion 상태/처리 결과 업데이트 |

## 7. 수용 기준

### 7.1 문의 등록
- 모든 필수 입력값이 올바르면 문의 등록이 성공해야 한다.
- 등록 성공 시 사용자는 성공 메시지를 확인할 수 있어야 한다.
- 잘못된 입력값이 있으면 등록 요청 이전에 오류가 표시되어야 한다.
- 서버 또는 워크플로우 오류가 발생하면 실패 메시지가 표시되어야 한다.

### 7.2 중복 처리
- 동일한 `이름 + 제목`의 기존 문의가 있으면 신규 문의는 저장되지 않아야 한다.
- 중복 등록 시 관리자 메일과 Notion 저장이 중복 발생하지 않아야 한다.
- 동시에 동일 문의가 여러 건 요청되더라도 최종적으로 1건만 등록되어야 한다.

### 7.3 관리자 처리
- 관리자는 문의 목록과 현재 상태를 조회할 수 있어야 한다.
- 관리자가 `처리중`으로 상태를 변경하면 저장소에 즉시 반영되어야 한다.
- 관리자가 `완료됨`으로 상태를 변경하면 완료 처리 워크플로우가 실행되어야 한다.
- 완료 처리 이후 문의자 메일, 관리자 완료 메일, Notion 상태 업데이트가 모두 반영되어야 한다.

## 8. 비기능 요구사항

- 주요 사용자 행동 결과는 명확한 성공/실패 메시지로 안내되어야 한다.
- 관리자 목록은 과제 범위 내에서 실사용 가능한 수준의 가독성을 제공해야 한다.
- 상태 변경과 워크플로우 호출은 중복 실행을 최소화해야 한다.
- 서비스 구조는 요구사항을 만족하는 범위에서 단순하고 이해 가능해야 한다.

## 9. 리스크 및 후속 결정 항목

- 동시성 환경에서의 중복 등록 방지 방식은 구현 단계에서 기술적으로 확정해야 한다.
- 관리자 보호 방식의 구체 구현은 기술 선택에 따라 달라질 수 있다.
- Notion DB 실제 속성명과 n8n 워크플로우 입출력 스키마는 구현 전에 고정해야 한다.
- 위 항목들이 장기 유지가 필요한 아키텍처 결정으로 굳어지면 별도 ADR로 관리한다.
