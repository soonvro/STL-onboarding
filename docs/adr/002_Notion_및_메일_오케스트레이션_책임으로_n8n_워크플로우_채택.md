1. **Title**
   Notion 및 메일 오케스트레이션 책임으로 n8n 워크플로우 채택

2. **Status**
   채택

3. **Context**
   과제 요구사항은 문의 등록 버튼 클릭 시 n8n 워크플로우를 호출해야 하고, 등록 워크플로우가 Notion DB 저장과 관리자 메일 발송을 수행해야 한다고 명시한다. 또한 완료 처리 시에도 n8n 워크플로우를 호출하고, 해당 워크플로우가 문의자 결과 메일, 관리자 완료 메일, Notion 상태 및 처리 결과 업데이트를 수행해야 한다. 따라서 외부 연동의 최종 실행 책임을 어디에 둘지 명확히 결정해야 한다.

4. **Decision**
   Notion 동기화와 메일 발송의 오케스트레이션 책임은 n8n 워크플로우가 가진다. 시스템은 `문의 등록`과 `문의 완료` 두 개의 n8n 워크플로우를 사용하고, 애플리케이션 백엔드는 입력 검증, 내부 상태 저장, 워크플로우 호출까지만 담당한다. Notion API 직접 호출과 메일 발송 로직은 애플리케이션 서버의 주 책임으로 두지 않는다.

   대안으로는 백엔드가 Notion과 메일을 직접 처리하는 방식이 있었지만, 과제 문구와 직접 충돌한다. 프론트엔드가 Notion 또는 메일 서비스와 직접 통신하는 방식도 고려 대상이 아니며, 비밀값 노출과 보안 문제 때문에 채택하지 않는다.

5. **Consequences**
   과제 요구사항과 시스템 책임 경계가 명확해진다. 외부 연동 흐름을 n8n 안에서 시각적으로 확인하고 수정하기 쉬우며, 등록과 완료 처리 로직을 워크플로우 단위로 분리할 수 있다. 반면 n8n이 별도 운영 컴포넌트가 되므로 가용성, 배포, 비밀값 관리가 추가된다. 또한 백엔드와 n8n 사이의 요청 형식, 재시도, 실패 처리 규약을 명확히 유지해야 한다.
