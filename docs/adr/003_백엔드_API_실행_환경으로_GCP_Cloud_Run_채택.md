1. **Title**
   백엔드 API 실행 환경으로 GCP Cloud Run 채택

2. **Status**
   채택

3. **Context**
   프론트엔드는 Vercel 정적 배포로 분리되므로, 문의 등록과 관리자 기능을 처리할 공개 HTTP API가 필요하다. 백엔드는 입력 검증, 중복 등록 방지, 내부 저장소 접근, 관리자 인증, n8n 호출을 담당해야 한다. 이번 과제에서는 인프라를 과도하게 늘리지 않으면서도 컨테이너 기반 배포, 환경 변수 및 비밀값 관리, 공개 엔드포인트 운영이 가능한 실행 환경이 적합하다.

4. **Decision**
   백엔드 API 실행 환경으로 GCP Cloud Run을 채택한다. 백엔드는 컨테이너 이미지 형태로 배포하고, 프론트엔드와 분리된 독립 서비스로 운영한다. Cloud Run은 공개 HTTP API 제공, 환경 변수 및 비밀값 주입, 관리형 확장 기능을 활용하는 방향으로 사용한다.

   대안으로는 가상 머신에 직접 배포하는 방식, 쿠버네티스 기반으로 운영하는 방식, 프론트엔드 플랫폼의 동적 서버 기능에 함께 올리는 방식이 있었다. 그러나 이번 과제 규모에서는 운영 복잡도 대비 이점이 작거나, 프론트/백엔드 경계를 흐리게 만들 수 있어 채택하지 않았다.

5. **Consequences**
   백엔드는 관리형 컨테이너 환경에서 단순하게 배포하고 운영할 수 있다. 프론트엔드와 배포 단위를 분리해 각자의 변경 주기를 독립적으로 가져갈 수 있고, 이후 데이터베이스나 n8n과의 연결도 별도 구성으로 확장하기 쉽다. 반면 Cloud Run 특성에 맞는 무상태 서비스 설계가 필요하고, 별도 도메인, 인증, 네트워크 연결 설정을 준비해야 한다. 또한 콜드 스타트와 서비스 간 경계 설정을 고려한 API 설계가 필요하다.
