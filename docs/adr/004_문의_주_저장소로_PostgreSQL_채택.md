1. **Title**
   문의 주 저장소로 PostgreSQL 채택

2. **Status**
   ADR 005 대체

3. **Context**
   과제는 Notion DB 저장과 업데이트를 요구하지만, 선택 과제는 Notion DB가 느릴 때도 동시에 같은 문의가 중복 등록되지 않도록 처리할 것을 요구한다. 또한 관리자 페이지는 문의 목록과 상태를 안정적으로 조회하고 변경할 수 있어야 한다. Notion API는 외부 시스템이므로 속도와 요청 제한의 영향을 직접 받으며, 동일한 `이름 + 제목` 조합의 동시 등록을 최종적으로 보장하는 주 저장소 역할까지 맡기기에는 제약이 크다. 따라서 문의 원본 데이터와 상태를 어디에 저장할지 별도 결정을 내려야 한다.

4. **Decision**
   문의의 주 저장소는 PostgreSQL로 한다. 문의 등록, 상태 변경, 처리 결과, 중복 방지 기준 데이터는 PostgreSQL에 먼저 기록하고, Notion은 n8n을 통해 동기화되는 외부 시스템으로 취급한다. 중복 등록 방지는 PostgreSQL의 제약 조건과 트랜잭션을 활용해 최종 보장한다.

   대안으로는 Notion만을 저장소로 사용하는 방식과 SQLite를 주 저장소로 사용하는 방식이 있었다. 그러나 Notion만으로는 동시성 상황에서 중복 등록 방지를 강하게 보장하기 어렵고, 관리자 조회와 상태 관리가 외부 API 성능에 직접 종속된다. SQLite는 단일 인스턴스 데모에는 단순하지만, 동시성 제어와 운영 확장성 측면에서 PostgreSQL보다 불리하다.

5. **Consequences**
   문의 데이터의 정본과 상태 전이를 내부에서 안정적으로 관리할 수 있고, 동시 중복 등록 방지 요구를 데이터베이스 수준에서 충족하기 쉬워진다. 관리자 페이지도 Notion 응답 속도에 직접 종속되지 않는다. 반면 Notion과 PostgreSQL 사이에 데이터 동기화 경계가 생기므로, 동기화 실패와 재시도 정책을 다뤄야 한다. 또한 운영 환경에 관계형 데이터베이스를 추가로 준비해야 한다.
