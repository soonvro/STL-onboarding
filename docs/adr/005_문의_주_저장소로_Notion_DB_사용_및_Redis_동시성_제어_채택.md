1. **Title**
   문의 주 저장소로 Notion DB 사용 및 Redis 동시성 제어 채택

2. **Status**
   채택

3. **Context**
   과제는 문의 등록과 완료 처리의 핵심 외부 연동을 n8n 워크플로우 안에서 수행하고, 등록과 완료 결과를 Notion DB에 반영하도록 요구한다. 또한 선택 과제는 Notion DB가 느릴 때도 동일한 `이름 + 제목` 조합의 문의가 동시에 중복 등록되지 않도록 처리할 것을 요구한다. 앞선 ADR 004는 PostgreSQL을 주 저장소로 두는 방향을 채택했지만, 현재 설계 논의에서는 Notion DB를 정본으로 유지하면서도 별도의 관계형 데이터베이스는 두지 않고 싶다는 제약이 생겼다. 따라서 Notion을 주 저장소로 사용하되, 동시성 제어를 어디에서 보완할지 새로 결정해야 한다.

4. **Decision**
   문의의 주 저장소는 Notion DB로 유지하고, Redis를 동시성 제어와 중복 방지 보조 저장소로 사용한다. 문의 등록 시 애플리케이션은 `이름 + 제목`으로부터 생성한 중복 판별 키를 기준으로 Redis에서 원자적 잠금과 상태 기록을 수행한 뒤 n8n 등록 워크플로우를 호출한다. n8n은 Notion 페이지 생성 시 같은 중복 판별 키를 함께 기록하고, 생성이 확인되면 Redis 상태를 확정 상태로 갱신한다. 상태 변경 역시 Notion 페이지 기준 잠금을 Redis에서 잡은 뒤 완료 워크플로우를 호출하는 방식으로 직렬화한다.

   대안으로는 PostgreSQL을 주 저장소로 사용하는 방식, Notion만 단독 저장소로 사용하는 방식, Redis를 주 저장소로 사용하는 방식이 있었다. PostgreSQL은 가장 단순하게 강한 제약 조건을 줄 수 있지만 별도 관계형 저장소 운영이 추가된다. Notion 단독 저장소는 선택 과제의 동시 중복 등록 방지를 강하게 보장하기 어렵다. Redis 단독 저장소는 과제의 Notion 중심 요구사항과 맞지 않고, 정본 저장소로 쓰기 위한 운영 부담이 커서 채택하지 않는다.

5. **Consequences**
   과제의 Notion 중심 요구사항을 유지하면서도 Redis의 원자적 연산을 이용해 동시 등록 경쟁 조건을 줄일 수 있다. 별도 관계형 데이터베이스 없이도 선택 과제의 핵심인 중복 등록 방지 요구를 설계상 반영할 수 있고, Redis 상태를 통해 등록 진행 중, 확정, 실패를 구분해 처리할 수 있다. 반면 정본은 Notion이고 동시성 제어는 Redis에 있으므로, 두 시스템 사이의 복구 절차가 필수다. Redis 잠금 만료, n8n 타임아웃, Notion 생성 후 응답 유실 같은 실패 상황에서는 중복 판별 키로 Notion을 재조회해 상태를 재조정해야 한다. 또한 관리자 목록과 상태 조회는 여전히 Notion API 성능과 제한에 영향을 받는다.
