1. **Title**
   프론트엔드 배포 전략으로 Vercel 정적 배포 채택

2. **Status**
   채택

3. **Context**
   이번 과제의 사용자 접점은 공개 문의 등록 페이지와 관리자 페이지다. 사용자는 브라우저에서 화면을 열고 입력과 조회를 수행하면 되며, Notion API 토큰, n8n 웹훅 키, 메일 발송 자격 증명처럼 비밀값이 필요한 연동은 브라우저에 노출되면 안 된다. 또한 현재 작업자는 정적 프론트엔드 배포 방식을 가장 익숙하게 다룰 수 있고, 구현 범위는 과제 요구사항을 충족하는 데 집중해야 한다. 따라서 프론트엔드는 렌더링과 사용자 입력 처리에 집중하고, 동적 처리와 보안 책임은 별도 백엔드에 두는 구성이 적합하다.

4. **Decision**
   프론트엔드는 정적 자산으로 빌드하고 Vercel에 배포한다. 브라우저는 정적 페이지와 자바스크립트만 내려받고, 문의 등록과 관리자 기능은 별도 백엔드 HTTP API를 호출해 수행한다. 정적 프론트엔드 안에 서버 렌더링, 서버 액션, 비밀값 기반 외부 연동 로직은 넣지 않는다.

   대안으로는 백엔드와 함께 단일 서버에서 프론트를 제공하는 방식, Vercel의 동적 서버 기능을 사용하는 방식이 있었다. 그러나 이번 과제에서는 정적 프론트로도 요구사항을 충족할 수 있고, 배포와 운영 복잡도를 낮추는 편이 더 중요하다고 판단했다.

5. **Consequences**
   프론트엔드 배포가 단순해지고, 정적 호스팅의 장점을 활용할 수 있다. 프론트엔드는 화면과 사용자 경험에 집중하고, 보안이 필요한 처리는 백엔드 경계 뒤로 분리할 수 있다. 반면 프론트와 백엔드가 서로 다른 배포 단위가 되므로 CORS, API 주소 관리, 관리자 인증 연동 같은 경계 설정이 필요하다. 또한 브라우저만으로는 외부 시스템 연동을 완결할 수 없으므로 백엔드 API가 필수 전제가 된다.
